# claudedash Comparison and Idea Harvest

## Scope Snapshot

**claudedash (local control plane)**
- Local-first observability for Claude Code sessions.
- Reads Claude Code artifacts on disk (`~/.claude/*`) and optional `.claudedash/*` plan files.
- Single-user, single-machine assumptions.
- Server + dashboard focused on visibility, context health, plan execution, and quality gates.
- No agent runtime orchestration; it observes and structures what Claude Code already does.

**forgemux (edge/hub orchestrator)**
- Orchestrates agent sessions (Claude Code, Codex CLI) via tmux and a session sidecar.
- Edge daemon + hub server + CLI with optional dashboard.
- Multi-node visibility, authentication, web attach, transcript capture, usage collection.
- More infrastructure heavy and multi-tenant-ready (single org).

## What claudedash Does That Forgemux Can Learn From

### 1) Plan Mode as a First-Class Operating Model
claudedash introduces a clear split between:
- **Live mode**: real-time visibility for sessions.
- **Plan mode**: structured task execution with dependencies, acceptance criteria, and status logging (`.claudedash/queue.md` + `.claudedash/execution.log`).

Forgemux explicitly excludes task queuing/scheduling in `docs/specs/roadmap.md`. However, a **lightweight plan execution format** could still be valuable as an optional overlay for:
- task-level auditability
- per-task quality gates
- structured postmortems

Idea: Treat plan mode as an optional **“project protocol”** rather than a scheduler. Forgemux can avoid owning task orchestration while still exposing plan insights in the dashboard.

### 2) Quality Gates as Structured Task Metadata
claudedash allows an agent to log lint/typecheck/test results per task into `execution.log` and surfaces them as a timeline. This is a clean interface:
- structured
- minimal required fields
- resilient to missing data

Idea: Add a `quality` event stream in Forgemux (per session or per task slice) so the dashboard can show “confidence signals” without needing to run CI itself.

### 3) Context Health as a Core Observability Signal
claudedash estimates context usage using session JSONL usage fields and highlights warnings at 65%/75%.

Forgemux already plans token tracking and usage collection (Phase 8), and includes references to Claude JSONL paths in `docs/specs/design.md`.

Idea: Promote **context pressure** to a top-level status signal (like “idle” or “waiting input”). This is higher UX impact than raw token counts.

### 4) Worktree Observability
claudedash detects and associates git worktrees with sessions, including:
- branch
- dirty status
- ahead/behind
- tasks mapped by cwd

Forgemux already creates worktrees for sessions. It should be able to expose:
- session → worktree mapping
- health signals per worktree
- branch divergence indicators to help reviewers

Idea: Add a Worktrees view in Forgemux dashboard with session/task association.

### 5) Compatibility Contracts for File-Based Integration
claudedash ships a `docs/compatibility.md` declaring the exact file paths and JSONL fields it depends on.

Forgemux intends to parse agent JSONL logs too (usage collector). A compatibility contract would reduce breakage when agent CLIs change.

Idea: Create `docs/compatibility.md` in Forgemux for Claude Code and Codex JSONL schemas + watched paths, and tie it to usage collectors.

### 6) Efficient Eventing and Caching Strategies
claudedash uses:
- a single shared SSE connection on the frontend
- mtime-based caching for file parsing
- watcher invalidation to avoid redundant re-reads

Forgemux will likely stream via WS and event stores. These are good patterns for the dashboard:
- minimize redundant connections
- de-duplicate events
- explicit cache invalidation signals

Idea: Design the Forgemux dashboard data layer around **single connection + event fanout**, even if the backend is more complex than file watching.

## Key Differences (and What They Imply)

1) **Observability vs. Orchestration**
- claudedash reads artifacts generated by a live agent.
- forgemux controls lifecycle and IO.

Implication: Forgemux can deliver deeper signals (TTY IO, process status, retries) but should still copy claudedash’s simplicity in its UX and data model where possible.

2) **Single-node vs Multi-node**
- claudedash: local only.
- forgemux: hub/edge model, cross-node aggregation.

Implication: Forgemux should be careful about costs of real-time UI updates; claudedash’s caching and “one connection” patterns can scale conceptually.

3) **Explicit Task Protocol**
- claudedash: plan protocol is optional but first-class in UX.
- forgemux: not planning to do task queues.

Implication: Adding an **optional task protocol** could be a differentiator without taking on scheduling responsibility.

## Forgemux Ideas Derived from claudedash

### Quick wins (documentation + UX)
1. Add `docs/compatibility.md` for agent JSONL formats and watched paths.
2. Add a **Context Health** badge on session cards (warn/critical levels).
3. Add a **Worktree panel** in the dashboard (branch, dirty, ahead/behind, mapped sessions).

### Medium-term product features
1. Introduce an **optional Plan Mode** spec:
   - `queue.md` + `execution.log` format (read-only by default).
   - Rendered as a dashboard view (no scheduling, no dispatch).
2. Add **Quality Gate events** (lint/typecheck/test results) and surface a per-task timeline.

### Deeper architectural bets
1. Define a **front-end event bus** with a single WS/SSE connection and local fanout.
2. Build an **adapter layer** between “agent log formats” and usage collectors, with resilience when fields are missing or renamed.

## Suggested Next Steps

1. Decide whether Forgemux wants an **optional plan protocol** (not a scheduler).
2. If yes, port claudedash’s minimal schema ideas into a Forgemux-compatible doc:
   - `docs/specs/plan-protocol.md`
3. Add **context health** as a first-class state alongside `Idle`, `WaitingInput`, etc.
4. Add a **worktree status** widget in the dashboard roadmap.
